Self-Hosted Agent Infrastructure: Lessons from Running OpenClaw

April 10, 2026 · Infrastructure · Agents

After months of running OpenClaw as my primary agent orchestration layer, I've gathered enough scars to share practical notes on self-hosted agent infrastructure. This isn't a setup guide — there are plenty of those. Instead: patterns that work, traps that don't, and why "just use the cloud" isn't always wrong.

The Promise and the Friction

Running agents locally sounds liberating: no API rate limits, no vendor lock-in, complete control. The reality is more nuanced. You're trading cloud convenience for operational complexity — and that trade only makes sense if you're actually hitting cloud constraints.

I started with a simple question: Can I run multiple specialized agents without paying per-token fees for orchestration logic? The answer is yes, but the architecture matters more than the hardware.

Architecture That Survives

The Gateway Pattern

My setup centers on a lightweight gateway daemon (the OpenClaw core) that handles routing, session state, and tool dispatch. Individual agents connect as clients, not servers. This inversion is critical — it means agents can be ephemeral, crash, or restart without the system losing coherence.

Key insight: State lives in the gateway, not in agents. Agents are stateless workers. This lets me kill and respawn agents mid-conversation without data loss. I've done this intentionally during long tasks when I wanted to switch models or clear context pollution.

Tool Registry Over Hardcoding

Early on, I hardcoded available tools into each agent. This failed quickly — every tool addition meant agent restarts. Now I maintain a dynamic tool registry in the gateway. Agents discover capabilities at runtime.

This decouples development: I can add a new data connector to the gateway and immediately use it from any agent without touching their code. The registry pattern is worth the initial complexity.

Session Persistence Without Bloat

Long-running agent conversations accumulate context. Without management, this becomes expensive (token usage) or breaks entirely (context window overflow).

My approach: progressive summarization. The gateway maintains full session logs, but agents receive only the current task + relevant context window + summary of prior work. Summaries are generated by a separate "compactor" agent that runs asynchronously. It's imperfect — occasionally misses nuances — but keeps the working agents efficient.

The Hard Parts

Model Context Protocols

Different local models (I run Ollama with 4-bit quantized variants) handle tool calling differently. Some follow OpenAI's function schema reliably. Others hallucinate parameters or ignore schemas entirely.

I ended up with per-model wrappers that normalize behavior. This is tedious maintenance — every new model needs testing and potential wrapper adjustments. There's no clean abstraction here; you adapt to what actually works.

Security Boundaries

Local agents have access to your actual system. This is the point, but also the risk. I run agents in constrained sandboxes where possible, but some tasks (infrastructure management, file operations) inherently need broader access.

My compromise: capability elevation through explicit gates. The gateway holds a capability manifest per agent. Sensitive operations require both the tool capability AND runtime approval (for interactive work) or pre-registered task signatures (for automated work). This isn't perfect security — it's managed risk.

Observability

Debugging multi-agent systems is harder than single agents. When something fails, was it the model, the tool, the prompt, or the orchestration logic?

I added structured logging throughout — every tool call, every model response, every routing decision. This creates noise, but I've learned to filter it. The alternative is guessing why a 5-step agent chain failed on step 4.

When This Makes Sense

Self-hosted agent infrastructure isn't for everyone. Consider it when:

  • Volume economics: You're running enough agent tasks that per-call API pricing hurts
  • Latency constraints: You need sub-second response times in tight loops
  • Data residency: Your use case can't send data to external APIs
  • Custom orchestration: Cloud platforms don't support your specific multi-agent patterns

For one-off tasks or simple chains, use the cloud. The operational overhead isn't worth it.

Current Stack

  • Orchestration: OpenClaw gateway + agent registry
  • Models: Ollama with quantized Llama and Qwen variants
  • Sessions: Elasticsearch for persistence + search
  • Tools: Mixed — some local (file system, shell), some remote (web search, translation APIs)
  • Hosts: Mac mini for development, VPS for persistent services

Open Questions

I'm still figuring out:

  • How to gracefully handle agent version upgrades mid-conversation
  • Whether to cache tool results more aggressively (currently conservative)
  • If there's a cleaner way to share context between agents without central bottlenecks

These are good problems — they mean the system is used enough to expose edge cases.

If you're building similar infrastructure, I'd be curious about your patterns. The space is still young enough that practical experience is more valuable than theoretical design.
