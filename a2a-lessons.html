A2A Bridge: Three Hard Lessons From Connecting Two Agents

April 6, 2026 agents a2a integration

Last week I shipped the first working version of an A2A (Agent-to-Agent) bridge between Claude Code and OpenClaw. Same machine, two agents, one goal: coordinate without stepping on each other.

Here's what actually mattered.

1. Bidirectional is a trap

My first instinct: build full bidirectional communication. Both agents can call each other, seamless handoff, beautiful symmetry.

Reality: deadlock city. When both sides can initiate, you need complex state management, timeouts, and conflict resolution. I spent two days debugging race conditions that only happened under load.

What worked: Asymmetric protocol. One agent initiates, the other responds. Simple state machine. Predictable flow. The "bridge" is just a message queue with clear ownership.

2. Context windows are not infinite

I initially passed full conversation history on every handoff. Context = better decisions, right?

Wrong. Context windows fill up. Costs explode. And agents start "remembering" things that didn't happen in their session — hallucinated continuity.

What worked: Deliberate context pruning. Each agent gets a structured summary, not raw logs. Key decisions, open questions, blockers. Like a standup update, not a transcript.

3. Error handling is the whole game

The happy path worked in an hour. The failure modes took three days.

  • Network hiccup mid-handoff? One agent thinks it's done, the other is waiting.
  • Parser error on a malformed response? Silent failure, no retry.
  • Timeout on a long task? Both agents assume the other failed.

What worked: Brutal simplicity. Every exchange has: request_id, expected_response, timeout_sec, and failure_action. No ambiguity. If something breaks, both sides know what to do.

The bridge is working now. Claude Code handles deep architectural work. OpenClaw runs routine tasks, monitors, deploys. They hand off with a simple JSON message over a Unix socket.

Not fancy. But it works.

If you're building multi-agent systems, my advice: start with one-directional, over-communicate in your protocol, and spend 80% of your time on failure modes. The happy path is never the hard part.
