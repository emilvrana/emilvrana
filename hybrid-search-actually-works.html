Hybrid Search Actually Works

June 9, 2026 · Emil Vrána

I rebuilt my RAG pipeline recently. The old one used pure semantic search — cosine similarity on embeddings, the standard approach everyone recommends. It worked fine for most queries.

Most isn't all. About 40% of real-world queries fell into a gap I hadn't accounted for.

The semantic search blind spot

Semantic search is great at finding conceptually related content. Ask "how do I handle authentication errors" and it finds the right docs even if they say "login failures" or "credential issues."

But it fails at two things people actually search for:

  • Exact identifiers. Error codes, function names, configuration keys, version numbers. Embeddings can't distinguish ERR_AUTH_401 from ERR_AUTH_403 — they look semantically identical.
  • Negation and specificity. "Not including deprecated fields" or "only version 2.1 endpoints." Semantic search averages meaning; it's bad at precise exclusion.

In a technical documentation corpus, these aren't edge cases. They're the majority of what people actually type.

What hybrid search adds

Hybrid search combines two retrieval strategies:

  • Semantic (dense): Embedding similarity — conceptually related content
  • Keyword (sparse): BM25 or similar — exact and partial term matching

The trick is the combination. Each covers the other's weaknesses.

For a query like "ERR_TIMEOUT in payment service v2," semantic search finds the right service context. Keyword search finds the exact error code. Together they find the specific paragraph that mentions both.

How to combine them

There are three common approaches, and only one I'd recommend for production:

1. Result fusion (Reciprocal Rank Fusion)

Run both searches independently, then merge results by rank position. Simple, works surprisingly well, and doesn't require tuning weights per query.

# RRF score for each document
def rrf_score(doc, rankings, k=60):
    score = 0
    for ranking in rankings:
        if doc in ranking:
            score += 1 / (k + ranking.index(doc))
    return score

The constant k smooths the impact of top positions. 60 is a solid default. You probably don't need to change it.

2. Weighted score merging

Normalize both score ranges, then combine with a weight like 0.7 * semantic + 0.3 * keyword.

This seems more precise but isn't. The weights depend on your data and query distribution, which means they're wrong for half your queries. I'd only use this if you've measured that a fixed weight genuinely beats RRF on your evaluation set.

3. Reranking (the one I recommend)

Run both searches with generous limits (top 30–50 each), merge and deduplicate, then run a cross-encoder reranker over the combined candidate set.

This is the most expensive option and the most effective. The reranker sees the full query-document pair and can judge whether a keyword match or a semantic match actually answers the question. It corrects both types of errors.

In my pipeline: pgvector for dense retrieval, BM25 via PostgreSQL full-text search for sparse, merge top 50 from each, then cross-encoder reranking. The whole thing runs in under 200ms on a single VPS.

What actually changed

On my evaluation set — 200 real queries against a technical documentation corpus:

  • Semantic-only: 61% recall@5
  • Hybrid with RRF: 78% recall@5
  • Hybrid with reranking: 84% recall@5

The 23-point jump from semantic-only to hybrid+rerank is almost entirely from queries involving exact identifiers, error codes, and version-specific content. Conceptual queries barely changed — semantic search was already handling those.

Why most tutorials skip this

Most RAG tutorials demonstrate on Wikipedia or general knowledge corpora. On those, semantic search works great because the queries are conceptual and the documents are prose-heavy.

Technical documentation is different. Code identifiers, configuration keys, and error codes are part of the content, and they need exact matching. If your users search for max_retries, they want that specific setting — not a paragraph about "retry strategies."

Practical setup

With PostgreSQL and pgvector, you already have both search types:

-- Semantic search (pgvector)
SELECT content, embedding <=> $1 AS distance
FROM documents
ORDER BY distance LIMIT 50;

-- Keyword search (PostgreSQL full-text)
SELECT content, ts_rank(to_tsvector('english', content), query) AS rank
FROM documents, plainto_tsquery('english', $1) query
WHERE to_tsvector('english', content) @@ query
ORDER BY rank DESC LIMIT 50;

Merge results, deduplicate, rerank with a small cross-encoder model. That's the whole thing.

No vector database needed. No separate search engine. PostgreSQL does both.
