The State Problem: Why Your Agent Forgets

July 16, 2026

Watch a production agent for a week and you'll notice something: the model rarely fails in isolation. What fails is state. The context that was supposed to be there but wasn't. The preference that was set three turns ago and silently dropped. The tool result that got truncated because something else consumed half the window.

We obsess over model quality — benchmarks, fine-tuning, prompt engineering. But the model is the easy part. The hard part is keeping the right state in front of it at the right time.

The Three Kinds of State

Every agent system deals with three distinct kinds of state, and conflating them is where most problems begin.

1. Conversation state

The message history. What the user said, what the agent replied, what tools were called. This is what the context window holds. Most teams think this is the only state that matters. It's not — it's the one that expires fastest.

2. Task state

Where we are in the current workflow. What step we're on, what's been completed, what's pending, what failed. This often lives in the agent's prompt as implicit context: "You are on step 3 of 5." When the prompt is truncated or rewritten, this state evaporates.

3. World state

Everything outside the conversation that the agent needs to know: user preferences, account settings, system status, recent events. This is the stuff that was true before the conversation started and will still be true after it ends. It's also the stuff most teams forget to inject consistently.

When an agent produces wrong output, the diagnosis usually looks like this: the model was fine, but it was working with stale conversation state, missing task state, or no world state at all.

The Context Window Is Not a Database

I've said this before, but it bears repeating: the context window is a transmission medium, not a storage layer. It's a pipe, not a tank.

The practical consequence: you cannot rely on the context window to maintain state across a long conversation. Not because the window is too small (it's often too large, which creates its own problems). Because the model processes the window fresh on every turn. There is no accumulated understanding. There is no learning within a session. There is only: "here's everything I've been given, what do I output now?"

This means your state management has to be external to the context window. The window carries state into the model. It does not store state for the model.

The Staleness Curve

State degrades predictably. The longer it sits without being refreshed, the less useful it becomes:

  • Conversation state: Useful for ~10-20 turns. After that, early turns add noise, not signal. The model re-reads old messages and over-weights them.
  • Task state: Useful until the task changes. After a pivot or error recovery, stale task state is actively harmful — the agent tries to resume a task that no longer applies.
  • World state: Useful until the world changes. This varies by domain — stock prices are stale in seconds, user preferences might be stable for months.

The staleness curve means your state management needs to be active, not passive. You don't just write state once and hope. You refresh, compress, and prune on every relevant event.

Practical State Management

Here's what actually works, based on building and running agents in production:

Separate storage from transmission

Keep your authoritative state in a proper data store — a database, a key-value store, a file. The context window is a view of that state, assembled fresh on every turn. Never read state from the context window and assume it's current. Read it from your store.

Summarize early, summarize often

Instead of letting conversation history grow until it hits the token limit, summarize proactively. After every 5-10 turns, compress the history into a structured summary: what was requested, what was done, what's pending, what failed. The summary replaces the raw history in the context window, not supplements it.

The key discipline: the summary must be structured, not narrative. "User asked for X, I did Y, result was Z" is a summary. "We discussed several options and the user seemed interested in the first one" is not — it's noise that the model will misinterpret.

Inject world state explicitly

Don't assume the model knows anything about the world it didn't learn in this conversation. If the user's timezone matters, inject it. If their subscription tier limits which actions they can take, inject that. If there's a known incident affecting the system, inject that too.

This injection should happen on every turn, not just the first one. Because the model doesn't remember the first turn — it re-processes the entire context window from scratch. If world state was only in turn 1, it's competing with 20 turns of conversation for attention.

Make task state explicit and mutable

Don't embed task state in the prompt as prose ("You are on step 3"). Make it a structured object that the agent can read and update:

task_state = {
  current_step: "validate_input",
  completed_steps: ["parse_request", "check_permissions"],
  failed_steps: [],
  pending: ["execute_action", "confirm_result"],
  retries: 0
}

This does two things: it makes state inspectable from outside the agent (you can log it, alert on it, debug it), and it makes it replaceable (you can reset, rollback, or fork task state without rewriting the entire context window).

The Litmus Test

Here's a simple test for whether your agent has a state problem:

Open a conversation. Have the agent do something non-trivial — three or four tool calls, a multi-step task. Then ask: "What did you just do, and why?"

If the answer is wrong, incomplete, or contradicts what actually happened, you have a state problem. The model isn't failing. The state it's working from is wrong.

Most agents fail this test after 10+ turns. Not because the model is bad at summarizing — because the state was never being managed properly in the first place.

What to Do Monday

  1. Audit your state surfaces. List every piece of state your agent relies on. Classify each as conversation, task, or world state. Note where it's stored and when it's refreshed. You'll find state that's never refreshed, state that's stored in the wrong place, and state that doesn't exist at all but the agent assumes it does.
  2. Move task state out of the prompt. Replace prose descriptions with structured state objects. Make them the single source of truth for where the agent is in a workflow.
  3. Add a state refresh step. Before every model call, explicitly refresh world state and validate task state. This costs a few milliseconds. It saves hours of debugging agent drift.
  4. Run the litmus test. After 10 turns, ask your agent what it did. If it can't tell you accurately, your state management needs work — not your model.

The model is the least of your problems. State is where agent systems actually live or die. Treat it like the critical infrastructure it is — not like an afterthought you'll "get to later."
