Decision Latency: The Hidden Cost of Thinking Too Much

July 10, 2026

Watch an AI agent "think" for 45 seconds to decide which of two tools to call. Watch it deliberate over a formatting choice. Watch it re-read the same context three times to answer a question it could have answered on the first pass.

This is decision latency — the time your system spends making choices that don't warrant deliberation. And it's killing your agent's usefulness far more reliably than any hallucination.

The Latency Problem Nobody Measures

Everyone tracks token cost. Almost nobody tracks decision latency — the cumulative time an agent spends on routing, tool selection, context assembly, and "thinking" before it produces output.

Here's what a typical agent request looks like in practice:

User asks a question
    ↓ 200ms — Input classification
    ↓ 800ms — Context retrieval (RAG)
    ↓ 1.5s — Agent "thinks" about which tool to use
    ↓ 2.0s — Model generates the tool call
    ↓ 500ms — Tool execution
    ↓ 1.2s — Agent "thinks" about what to do with the result
    ↓ 800ms — Final response generation
    = 7 seconds total

Of those 7 seconds, 3.7 seconds — more than half — are the model deliberating over decisions that are, in most cases, deterministic. The retrieval took 800ms. The tool execution took 500ms. The "thinking" added 3.7 seconds of latency for zero reliability gain.

This compounds brutally. An agent that makes 5 sequential tool calls doesn't just pay this tax once — it pays it on every loop iteration. A 35-second interaction where 20 seconds is model deliberation about routine decisions is 20 seconds your user spends watching a spinner.

Fast Path, Slow Path

The solution isn't to make the agent less intelligent. It's to recognize that most decisions don't require intelligence at all.

Google has been doing this for decades. Search queries go through a fast path — cache hits, spelling corrections, query classification — before they ever hit the ranking model. The model does the hard work. The fast path handles everything else in milliseconds.

Your agent should work the same way:

Fast path: deterministic routing

  • Input classification: Rule-based. "This looks like a search query" → retrieval pipeline. "This looks like a computation" → code execution. "This looks like a formatting request" → template renderer. No model call needed.
  • Tool selection: If you have 3 tools, you don't need a model to pick one. You need a routing function — a simple classifier, keyword match, or intent tag.
  • Context assembly: Most context is static or semi-static. Pre-assemble it. Cache it. Don't reconstruct it on every request.

Slow path: model deliberation

  • Novel reasoning: Questions the classifier hasn't seen before.
  • Ambiguous cases: Where multiple tools could apply and the tradeoffs are real.
  • Multi-step planning: Where the agent genuinely needs to reason about sequencing.

The trick is making the fast path cover as much ground as possible. Every request that hits the slow path is an opportunity to learn a new fast path rule.

The Caching Principle

If your agent sees the same type of request twice and handles it the same way both times, the second time should be faster. Not because the model is faster, but because you learned from the first time.

This is the caching principle applied to decisions:

  1. Observe: Log every decision the model makes — routing, tool selection, output format.
  2. Pattern: After N observations of the same pattern, extract it into a rule.
  3. Automate: Route matching requests through the rule, bypassing the model entirely.
  4. Monitor: Track accuracy. If the rule degrades, fall back to the model and re-learn.

This is how you go from an agent that takes 7 seconds per request to one that takes 1.5 seconds for 80% of requests and 5 seconds for the remaining 20% that actually need model intelligence.

Decision Budget

I use a concept I call the decision budget — the maximum number of non-trivial model decisions allowed per request. Not token budget. Decision budget.

A request to "summarize this document" gets a budget of 1: one model call to produce the summary. No tool selection deliberation. No "should I retrieve more context?" No "let me re-read the document to make sure." One call. If the result isn't good enough, the fallback is a more expensive second pass — but the first pass should be fast and cheap.

A complex research query gets a budget of 3-5: route, retrieve, synthesize, verify, respond. Each step is deliberate. Each step is justified by the complexity of the task.

The budget forces you to think about which decisions matter. And most decisions, it turns out, don't.

The UX of Speed

Here's the thing about latency: users don't experience it as "the model is thinking." They experience it as "this thing is broken."

The 3-second rule from web UX applies to agents too. Under 3 seconds feels responsive. 3-10 seconds feels slow but acceptable. Over 10 seconds and your user is reaching for the close button, no matter how brilliant the answer.

This isn't about making agents faster for the sake of benchmarks. It's about making them usable. An agent that delivers a good answer in 2 seconds beats an agent that delivers a perfect answer in 15 seconds — because in 15 seconds, the user has already left, tabbed away, or asked the question somewhere else.

Speed is a feature. Not the only feature, but the one your users notice first.

What to Do Monday

  1. Instrument decision latency. Add timing around every model call. Not just total response time — time per decision point within the agent loop.
  2. Identify the fast path. Look at your last 1000 requests. How many followed the same routing pattern? Those are your fast path candidates.
  3. Replace deliberation with rules. For the top 3 routing decisions, replace model calls with deterministic classifiers. Measure the accuracy difference. I'll bet it's smaller than you think.
  4. Set a decision budget. Cap the number of model turns per request type. Force yourself to get the same result with fewer decisions.

Your agent doesn't need to think about everything. The most reliable systems make fast, local decisions and save deliberation for where it actually matters. Stop paying the latency tax on choices that should be free.
