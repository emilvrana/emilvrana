Multi-Agent Patterns: When One Agent Isn't Enough

April 4, 2026 · Emil Vrána

Most agent tutorials stop at the single-agent case. You build a chatbot, a code assistant, a RAG system. It works in isolation. But real production systems rarely run alone — they integrate with other agents, human workflows, and existing infrastructure.

I've spent the last few weeks building an A2A (Agent-to-Agent) bridge between Claude Code (running Opus 4.6) and OpenClaw (my local agent system). The goal was simple: let the two systems delegate tasks to each other based on their strengths. What I found is that the hard part isn't the protocol — it's deciding when to hand off, what to pass along, and how to maintain continuity.

The case for multiple agents

A single agent has a fixed context window, a fixed knowledge cutoff, and fixed capabilities. It can't simultaneously be great at everything. Claude Code (Opus) is excellent at complex reasoning and architecture but expensive and slower. Sonnet is fast and cheap but occasionally makes architectural mistakes. My local system (OpenClaw) has tool access, long-term memory, and can run indefinitely — but it's running a smaller model and struggles with complex multi-step reasoning.

The obvious solution: route tasks to the agent best suited for them. But this isn't as simple as "hard tasks go to Opus, easy tasks go to Sonnet." Task complexity isn't always obvious upfront. And sometimes a task starts simple and gets complicated, or vice versa.

What I built

The bridge is a lightweight protocol over HTTP. Each agent exposes a /delegate endpoint that accepts:

  • task: A description of what needs to be done
  • context: The relevant background — not the full conversation history, just what matters
  • artifacts: Files, code snippets, any concrete inputs
  • constraints: Hard limits (time, cost, what not to do)
  • callback: Where to send the result

The receiving agent can either accept the task or escalate it further. Each delegation is logged with a unique ID, so you can trace the chain later.

The patterns that emerged

1. Router agent pattern

One agent acts as a dispatcher. It doesn't do the work itself — it just classifies incoming requests and routes them. This works well when you have a clear taxonomy of tasks and your routing logic is stable.

{"complexity": "high", "domain": "architecture", "suggested_handler": "opus"}
{"complexity": "low", "domain": "infrastructure", "suggested_handler": "openclaw"}

The downside: the router can become a bottleneck, and its classification errors are expensive (wrong routing means either overpaying for simple tasks or getting poor results on hard ones).

2. Escalation pattern

Start with the cheap/fast agent. If it fails confidence checks or hits complexity walls, escalate. This is what I'm using now:

  • OpenClaw (Sonnet) tries the task first
  • If it detects complexity > threshold, it escalates to Claude Code
  • Claude Code can delegate back for tool execution

The key is making escalation cheap. You don't want to burn 10k tokens before realizing you need Opus. I use a fast pre-flight check: "Estimate the complexity of this task on a scale of 1-5 and explain why." This costs ~500 tokens and catches most mis-routings early.

3. Specialist pool pattern

Instead of generalist agents, you have specialist agents: one for code review, one for infrastructure, one for writing. The router selects from a registry based on task type. This is clean in theory but I've found it hard to maintain — the boundaries between specialists blur over time, and you end up with overlapping capabilities.

The hard parts

Context continuity

When Agent A hands off to Agent B, what does B need to know? Not everything. I've learned to pass:

  • The immediate task
  • Key decisions already made
  • Current constraints (budget, time, scope)
  • What's explicitly out of scope

What I don't pass: the full conversation history, every exploratory thought, tool call logs. These bloat the context and often confuse more than they help.

Tool access asymmetry

Claude Code in a sandbox can't exec commands on my local machine. OpenClaw can. This means some tasks require handoff not because of complexity, but because of capability gaps. I've had to design the protocol so agents can advertise their capabilities:

{"capabilities": ["file_read", "file_write", "shell_exec", "web_search", "git"],
 "limitations": ["no_internet", "no_github_write_without_pat"]}

Cost accounting

With multiple agents, you need to track costs across the chain. I propagate a cost_budget_remaining field through delegations. If Agent A has $0.10 left and delegates to Agent B, B knows it can't spend $0.50 on its own.

Error handling

When an agent fails, who handles it? The original requester? The agent that delegated? I've settled on this: if Agent B fails, it reports back to Agent A with an error classification. A decides whether to retry, escalate, or fail the whole chain.

What actually works

After two weeks of experimentation, here's what's in production:

  • Escalation over routing: Start cheap, escalate when needed. The overhead of a router agent isn't worth it for my volume.
  • Minimal context passing: Summarize, don't forward. The receiving agent should get what it needs, not what happened.
  • Explicit capability negotiation: Agents declare what they can and can't do before accepting tasks.
  • Structured handoff protocol: JSON schema for tasks, not natural language. Less ambiguity.

When multi-agent makes sense

Not every system needs this. If your tasks are uniform in complexity and your single agent handles them well, adding multi-agent coordination is just overhead. But consider it when:

  • You have clear capability differences (e.g., local tools vs. cloud reasoning)
  • Cost matters and you can classify tasks accurately
  • Tasks vary widely in complexity
  • You need 24/7 operation but can't afford Opus for everything

The pattern I'm happiest with: treat agents as services with different SLAs. The strong reasoner is your premium tier. The fast local agent is your default. Route between them based on what's at stake.

I'm currently running this setup for my own work — Claude Code for complex design and debugging, OpenClaw for routine tasks and anything needing local tool access. The bridge adds ~100ms of latency per handoff, which is acceptable for the cost savings on routine work.
